# 1️⃣ Where should `accept()` live?

You noticed a contradiction:

* I showed `accept(listenFd, ...)`
* But earlier I said `Socket` should have `accept()`

Let’s fix this cleanly.

---

## 🔵 What is `accept()` really?

`accept()` operates on a **listening socket fd**.

So logically:

* It belongs to the **listening socket**
* Not to a client connection
* Not to Connection class

---

## ✅ Correct Design

### `Socket` (Listening socket only)

```cpp
class Socket
{
private:
    int _fd;  // listening fd

public:
    void create();
    void bind();
    void listen();
    int acceptClient();   // wraps ::accept()
    int getFd() const;
};
```

### Implementation:

```cpp
int Socket::acceptClient()
{
    int clientFd = ::accept(_fd, NULL, NULL);
    return clientFd;
}
```

---

## 🔥 So who calls accept?

`ConnectionManager` should NOT call raw `::accept()`.

Instead:

```cpp
void ConnectionManager::handleNewConnection(int listenFd)
{
    Socket& sock = _listenSockets[listenFd];

    int clientFd = sock.acceptClient();
    setNonBlocking(clientFd);

    Connection* conn = new Connection(clientFd);
    _connections[clientFd] = conn;

    _eventLoop->addFd(clientFd, EPOLLIN);
}
```

---

## 🚫 Why NOT put accept inside Connection?

Because:

* A `Connection` represents an **already accepted client**
* It does not create itself
* It should not know about listening sockets

That would violate responsibility separation.

# 2️⃣ What is `isRequestComplete()`?

Very important concept.

Remember:

* TCP is a stream
* `recv()` may return:

  * Half request
  * One request
  * One and a half requests

So your server reads data gradually.

---

## Example

Client sends:

```
GET /index.html HTTP/1.1\r\n
Host: localhost\r\n
Content-Length: 5\r\n
\r\n
Hello
```

You might receive it in 3 chunks:

1. `"GET /index.html HTTP/1.1\r\nHost: lo"`
2. `"calhost\r\nContent-Length: 5\r\n\r\nHe"`
3. `"llo"`

---

## So what does `isRequestComplete()` mean?

It means:

> Has the full HTTP request been received and parsed?

---

## Inside Connection

```cpp
class Connection
{
private:
    bool _requestComplete;
};
```

---

## In handleRead()

```cpp
void Connection::handleRead()
{
    char buffer[4096];
    int bytes = recv(_fd, buffer, sizeof(buffer), 0);

    _readBuffer.append(buffer, bytes);

    if (_parser.parse(_readBuffer, _request))
        _requestComplete = true;
}
```

---

## So `isRequestComplete()` is simply:

```cpp
bool Connection::isRequestComplete() const
{
    return _requestComplete;
}
```

---

# 🔥 Why is this needed?

Because in epoll:

* You get EPOLLIN events
* But that does NOT mean the request is complete
* It only means: "there is data available"

So you must:

1. Keep reading
2. Keep parsing
3. Only build response once full request is ready

---

# 🔥 Clean Mental Model

## Listening socket

Accepts clients.

## Connection

Handles one client:

* owns read buffer
* owns parser
* owns request
* owns response
* knows when request is complete

## ConnectionManager

Coordinates everything.

---

# 🔥 Bonus: Real Flow with isRequestComplete()

```
EPOLLIN →
    Connection::handleRead()
        append data
        parse
        if complete → set flag

ConnectionManager sees:
    if conn.isRequestComplete():
        build response
        switch to EPOLLOUT
```

# 🎯 Final Clean Responsibility Table

| Class             | Responsibility                 |
| ----------------- | ------------------------------ |
| Socket            | Listening socket only          |
| Connection        | One client                     |
| ConnectionManager | Owns all connections           |
| EventLoop         | Detects events                 |
| RequestParser     | Knows when request is complete |
