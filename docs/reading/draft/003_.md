# 🔌 1️⃣ `Socket.hpp`

### ❓ Why not just use raw int file descriptors?

You *can*.
But wrapping them gives you:

* cleaner code
* safer ownership
* easier testing
* less repetition

---

### What It Usually Contains

```cpp
class Socket {
public:
    Socket();
    ~Socket();

    void create();
    void bind(const std::string& ip, int port);
    void listen(int backlog);
    int accept();

    void setNonBlocking();

    int getFd() const;

private:
    int _fd;
};
```

---

### Why It’s Useful

Instead of:

```cpp
int fd = socket(...);
fcntl(fd, ...);
bind(fd, ...);
```

You write:

```cpp
Socket sock;
sock.create();
sock.setNonBlocking();
```

It keeps **core networking logic clean**.

---

# 🗂 2️⃣ `ServerConfig.hpp`

This is VERY important.

This file is pure data.

It represents one:

```nginx
server {
    ...
}
```

---

### Example

```cpp
class ServerConfig {
public:
    std::vector<std::pair<std::string, int> > listens;
    std::vector<std::string> server_names;
    std::string root;
    size_t client_max_body_size;

    std::vector<LocationConfig> locations;
};
```

---

### Why It Exists

Because once parsing is done, you need structured storage of:

* listen pairs
* root
* error pages
* locations

This object is:

> The in-memory representation of your nginx server block.

Without it, your routing layer has nothing clean to work with.

---

# 📝 3️⃣ `Logger.hpp`

### Do You NEED it?

No.

### Should You have it?

YES.

Because debugging epoll + HTTP without logging is hell.

---

### Example

```cpp
class Logger {
public:
    static void info(const std::string& msg);
    static void error(const std::string& msg);
    static void debug(const std::string& msg);
};
```

---

### Why It's Useful

Instead of:

```cpp
std::cout << "Error: " << msg << std::endl;
```

You centralize:

* formatting
* debug toggling
* log levels
* maybe colored output

And later you can disable debug logs easily.

---

# 📁 4️⃣ `FileUtils.hpp`

This one is extremely useful for Webserv.

Because you will repeatedly need:

* file exists?
* is directory?
* read file into string
* get file size
* get extension
* build path safely

---

### Example

```cpp
class FileUtils {
public:
    static bool exists(const std::string& path);
    static bool isDirectory(const std::string& path);
    static std::string readFile(const std::string& path);
    static size_t fileSize(const std::string& path);
};
```

---

### Why It’s Important

Otherwise you will have:

```cpp
struct stat st;
stat(path.c_str(), &st);
```

Scattered in 20 places.

Centralizing filesystem logic prevents duplication and bugs.

# 🧠 PART 2 — Real Dependency Arrows Between Folders

This is CRITICAL.

Let’s define allowed dependencies.

---

## 📊 Dependency Rules

```
main
  ↓
core
  ↓
http
  ↓
routing
  ↓
config
  ↓
utils
```

But that’s too linear. Let’s draw real relationships.

---

# 🔥 Clean Dependency Graph

```
           utils
             ↑
config   http   core
   ↑       ↑       ↑
     routing
        ↑
       main
```

Now let’s clarify properly.

---

## utils/

Lowest level.

Everyone can depend on utils.

utils depends on nobody.

---

## config/

Can depend on:

* utils

Must NOT depend on:

* http
* routing
* core

Because config is static parsing only.

---

## http/

Can depend on:

* utils

Must NOT depend on:

* routing
* core
* config (ideally)

HTTP parsing should not know about server config.

---

## routing/

Can depend on:

* config
* http
* utils

Because routing needs:

* HttpRequest
* ServerConfig
* LocationConfig

Must NOT depend on:

* core

Routing should not know about sockets.

---

## core/

Can depend on:

* http
* routing
* config
* utils

Because it orchestrates everything.

But:

* http must not depend on core.
* config must not depend on core.

Core is infrastructure that *uses* higher layers, not the opposite.

---

# 🧠 Golden Rule

Dependencies should flow **downwards toward more generic layers**, not upwards.

---

# 🚫 What Creates Circular Dependencies?

Example of BAD design:

If:

```
LocationConfig includes Router.hpp
```

and

```
Router includes LocationConfig.hpp
```

Boom 💥 circular dependency.

---

# 🧩 PART 3 — How Includes Should Look

Now let's be concrete.

---

# 📌 Rule 1 — Use Forward Declarations

Instead of:

```cpp
#include "HttpRequest.hpp"
```

You can sometimes do:

```cpp
class HttpRequest;
```

If you only use pointer or reference.

---

# 📌 Rule 2 — Include in .cpp When Possible

In header:

```cpp
class HttpRequest;
class ServerConfig;

class Router {
public:
    HttpResponse route(const HttpRequest& req,
                       const std::vector<ServerConfig>& servers);
};
```

Then in `Router.cpp`:

```cpp
#include "Router.hpp"
#include "HttpRequest.hpp"
#include "ServerConfig.hpp"
```

This prevents circular include chains.

---

# 📌 Rule 3 — Never Include Core Inside Config

Wrong:

```cpp
// ServerConfig.hpp
#include "Socket.hpp"   ❌
```

Config should not know about sockets.

---

# 📌 Rule 4 — Avoid Including Whole Layers

Instead of:

```cpp
#include "../core/EventLoop.hpp"
```

Depend only on minimal required classes.

---

# 🎯 Clean Include Example Per Folder

---

## config/LocationConfig.hpp

```cpp

```

No routing.
No http.

```cpp

#include "HttpRequest.hpp"
#include "HttpResponse.hpp"
```
